# Lab6 操作系统答辩文档

## 1. 大致说一下 Lab6 的调度算法的逻辑

Lab6 主要实现了 **Round Robin (RR, 时间片轮转)** 调度算法，并作为扩展（Challenge）介绍了 **Stride (步长)** 调度算法。

*   **调度框架**：Lab6 引入了 `sched_class` 接口，将具体的调度策略与内核的调度机制解耦。内核在适当时机（如进程创建、结束、时间片耗尽）调用 `sched_class` 中的函数（如 `enqueue`, `dequeue`, `pick_next`）。

*   **Round Robin (RR) 逻辑**：
    1.  **维护队列**：使用一个双向链表 (`run_list`) 作为运行队列，维护所有 `RUNNABLE` 状态的进程。
    2.  **入队 (Enqueue)**：新创建或被唤醒的进程加入队列尾部。
    3.  **调度 (Pick Next)**：每次调度时，从队列头部选择进程运行，体现 FIFO (先进先出) 原则。
    4.  **时间片 (Tick)**：在时钟中断中递减当前进程的 `time_slice`。
    5.  **切换**：当 `time_slice` 减为 0 时，设置 `need_resched` 标志。调度器将该进程放到队列尾部，并重置其时间片，然后选择队头的新进程运行。

*   **Stride (步长) 调度逻辑** (扩展)：
    1.  **优先级**：为每个进程分配一个优先级 (priority)。
    2.  **步长 (Stride)**：计算 `stride = BIG_STRIDE / priority`。优先级越高，步长越小。
    3.  **进度 (Pass)**：每个进程有一个 `pass` 值，表示当前的调度进度。
    4.  **调度决策**：每次从运行队列中选择 `pass` 值**最小**的进程运行。
    5.  **更新**：进程运行一个时间片后，更新其 `pass` 值：`pass += stride`。这意味着优先级高的进程 `pass` 增加得慢，会被更频繁地调度。
    6.  **数据结构**：为了高效查找最小 `pass`，通常使用**斜堆 (Skew Heap)** 或优先队列来管理运行队列。

## 2. 和 Lab5 相比，proc struct 增加了哪些字段，都是什么含义

在 `kern/process/proc.h` 中，`struct proc_struct` 增加了以下字段以支持 Lab6 的调度算法：

| 字段名 | 类型 | 含义 | 作用 |
| :--- | :--- | :--- | :--- |
| `rq` | `struct run_queue *` | 运行队列指针 | 指向该进程所属的运行队列，方便进程查找自己所在的队列。 |
| `run_link` | `list_entry_t` | 运行队列链表节点 | 用于将进程连接到 `run_queue` 的双向链表中 (主要用于 RR 算法)。 |
| `time_slice` | `int` | 时间片 | 记录该进程当前剩余的可运行时间片大小 (tick 数)。 |
| `lab6_run_pool` | `skew_heap_entry_t` | 斜堆节点 | 用于 Stride 调度算法，作为斜堆（优先队列）的节点。 |
| `lab6_stride` | `uint32_t` | 当前步长/进度 (pass) | Stride 算法中的 pass 值，调度器根据此值选择最小者运行。 |
| `lab6_priority` | `uint32_t` | 优先级 | Stride 算法中的优先级，用于计算步长增量。 |

## 3. 当前整体的调度框架，ucore 单核，这对 run_queue 结构有什么影响

*   **全局唯一的运行队列**：由于 uCore Lab6 是单核操作系统，同一时刻只能有一个 CPU 在执行指令，因此只需要维护**一个全局的 `struct run_queue`** (在代码中通常是 `__rq`)。
*   **简化同步**：在单核环境下，为了保护 `run_queue` 不被并发访问破坏（主要是防止中断处理程序和内核线程之间的竞争），通常只需要在访问队列前**关闭中断 (`local_intr_save`)**，访问结束后恢复中断即可。不需要复杂的自旋锁 (Spinlock) 或其他多核同步机制。
*   **结构简单**：`run_queue` 结构体中不需要包含 CPU ID 或每 CPU 的私有数据区。

## 4. 主动调度和被动调度的区别

*   **主动调度 (Voluntary Scheduling)**：
    *   **定义**：进程在运行过程中，因为需要等待某个外部事件（如 IO 操作完成、等待锁、等待子进程结束）或者自身任务完成（退出），**主动调用** `schedule()` 函数放弃 CPU。
    *   **场景**：`do_yield` (让出 CPU), `do_wait` (等待子进程), `do_exit` (进程退出), `down` (信号量等待), IO 阻塞。
    *   **状态变化**：进程状态可能从 `RUNNABLE` 变为 `SLEEPING` (等待状态) 或 `ZOMBIE`。

*   **被动调度 / 抢占调度 (Involuntary / Preemptive Scheduling)**：
    *   **定义**：进程本来正在运行且愿意继续运行，但由于外部原因（如时间片耗尽、更高优先级的进程就绪），被调度器**强制剥夺** CPU 使用权。
    *   **场景**：时钟中断中发现 `time_slice == 0`；唤醒一个高优先级进程时。
    *   **机制**：通常通过设置 `current->need_resched = 1` 标志位。在中断或异常处理即将返回用户态时，检查该标志位，如果为 1 则调用 `schedule()`。
    *   **状态变化**：进程状态由 `RUNNABLE` (Running) 变为 `RUNNABLE` (Ready, 也就是放回队列等待下次调度)。

## 5. 内核态为什么是主动调度

*   **简化内核设计**：在 uCore 及许多简单 OS 的设计中，内核代码（系统调用、异常处理）执行期间通常被设计为**不可抢占** (Non-preemptible) 或者**协作式调度**。
*   **数据一致性与锁**：内核中包含大量共享数据结构（如进程链表、页表等）。如果允许内核态代码在任意时刻被抢占（被动调度），那么内核必须在几乎所有关键数据结构访问上加锁，这将极大地增加内核的复杂度和死锁风险。
*   **设计约定**：只有当内核代码显式调用 `schedule()`（即主动调度，例如等待磁盘 IO）时，才认为当前状态是安全的，可以进行上下文切换。
*   **注**：现代通用操作系统（如 Linux 2.6+）支持内核抢占，但需要更复杂的锁机制。uCore Lab6 作为一个教学 OS，通常只在用户态边界进行抢占检查。

## 6. schedule_class 的作用

`schedule_class` 结构体是调度器框架的核心抽象，其作用如下：

*   **接口抽象**：它定义了一组通用的函数指针接口（`init`, `enqueue`, `dequeue`, `pick_next`, `proc_tick`），描述了一个调度算法应该具有的行为。
*   **解耦**：它将**调度机制**（`schedule()` 函数，负责切换上下文的通用逻辑）与**调度策略**（具体的算法，如 RR 或 Stride）分离开来。`kern/schedule/sched.c` 中的代码不需要知道具体在用什么算法。
*   **运行时替换**：通过修改全局指针 `sched_class` 的指向，可以在系统运行时（或编译时）轻松切换不同的调度算法，而无需修改核心调度代码，极大地提高了系统的可扩展性。

## 7. run_decode 冲突了怎么办

*(注：此处 "run_decode" 疑为 **"Stride 溢出"** 或 **"Run Queue 竞争"** 的笔误，以下主要针对 Stride 调度中经典的数值溢出问题进行回答)*

在 Stride 调度算法中，最经典的冲突与数值问题是 **Stride/Pass 值的溢出**：

*   **问题**：`pass` 和 `stride` 是无符号整数（如 `uint32_t`）。随着进程运行，`pass` 值会不断增加，最终会超过 32 位整数的最大值并发生回绕（溢出）。如果直接比较绝对大小，溢出后的较小值会导致优先级错误的判断。
*   **解决方法**：利用无符号整数减法的性质。
    *   不直接比较 `pass_a < pass_b`。
    *   而是比较 `(int32_t)(pass_a - pass_b) < 0`。
    *   只要两个进程的 `pass` 值差距不超过整数范围的一半（`BIG_STRIDE` 的选取保证了这一点），即使发生溢出回绕，减法得到的结果转换成有符号整数后，也能正确反映两者的先后顺序（环形空间比较）。

*(如果问题是指 **Run Queue 数据结构操作冲突**)*：
*   **解决方法**：在单核 uCore 中，通过**关中断 (`local_intr_save`)** 来防止中断处理程序打断当前的队列操作，从而避免数据不一致。

## 8. 多核情况下怎么解决

如果将 uCore 扩展到多核 (SMP)，调度器面临的主要挑战是 `run_queue` 的并发访问：

1.  **Per-CPU Runqueue (每 CPU 运行队列)**：
    *   **方案**：不再使用全局单一的 `run_queue`，而是为每个 CPU 分配一个独立的 `run_queue`。
    *   **优势**：每个 CPU 优先操作自己的队列，绝大多数调度操作不需要竞争全局锁，极大提高了并行性。

2.  **锁机制 (Locking)**：
    *   虽然有了 Per-CPU 队列，但仍需**自旋锁 (Spinlock)** 保护。因为一个 CPU 可能需要访问另一个 CPU 的队列（例如负载均衡时），或者中断处理程序可能在不同核上运行。
    *   操作队列时需：关中断 + 获取自旋锁。

3.  **负载均衡 (Load Balancing)**：
    *   **问题**：可能出现 CPU A 的队列有很多进程，而 CPU B 空闲的情况。
    *   **解决**：实现负载均衡机制（如 "Work Stealing" 任务窃取，或周期性的 "Push Migration"），将进程从繁忙的 CPU 迁移到空闲的 CPU。这需要跨 CPU 操作队列，需要持有目标队列的锁。

4.  **CPU 亲和性 (Affinity)**：
    *   调度器应尽量让进程保持在同一个 CPU 上运行，以利用 CPU 缓存（L1/L2 Cache）的局部性，减少 Cache Miss。
