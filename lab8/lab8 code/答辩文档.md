# Lab 8 答辩与练习文档

## 问答题

### 1. 为什么要有文件系统
文件系统主要提供了数据的持久化存储、组织和管理机制。
*   **持久化**：使数据在进程结束后甚至掉电后依然存在。
*   **组织管理**：通过目录和文件名的层级结构（命名空间）来组织数据，方便用户和程序访问，而不是直接操作磁盘扇区。
*   **抽象**：将底层的块设备（Block Device）抽象为逻辑上的文件（File）和目录（Directory）。
*   **保护与共享**：提供权限控制，允许多个进程/用户安全地共享或隔离数据。

### 2. sfs_inode 和 sfs_disk_inode 的区别
*   **sfs_disk_inode (On-Disk Node)**:
    *   **位置**：存储在磁盘上。
    *   **内容**：包含文件的元数据，如文件大小、类型、硬链接数、数据块索引（直接/间接指针）。
    *   **用途**：文件的持久化描述。纯数据结构，不包含运行时状态。
*   **sfs_inode (In-Memory Node)**:
    *   **位置**：存储在内存中（内核堆）。
    *   **内容**：包含 `sfs_disk_inode` 的指针（或副本），以及**运行时特有的信息**，如信号量（`sem`，用于同步互斥）、脏位（`dirty`，表示是否修改过）、引用计数（`reclaim_count`）、链表节点（用于由 SFS 管理 inode 列表）等。
    *   **用途**：文件系统运行时操作的对象。

### 3. 文件系统怎么处理设备
虽然文件系统主要管理文件，但在 UNIX/uCore 设计中，“一切皆文件”。
*   **抽象**：设备被抽象为特殊的 inode。通过 VFS（虚拟文件系统）层统一接口。
*   **接口**：VFS 定义了通用的 inode 操作（`vop_read`, `vop_write` 等）。对于设备文件，这些操作被映射到具体的设备驱动函数。
*   **实现**：`driver/` 目录下实现了具体的设备驱动。`fs/devs/` 下实现了将设备包装为 device 结构体的逻辑（如 `dev_stdin`, `dev_stdout`）。文件系统通过 `vfs_add_dev` 将设备注册为一个名称（如 "stdin"），并为其创建 inode。访问该 inode 时，实际上调用的是 `struct device` 中的函数指针。

### 4. 设备之间使用什么数据结构
主要使用 `struct device` (定义在 `kern/fs/devs/dev.h`)：
```c
struct device {
    size_t d_blocks;          // 设备块数量
    size_t d_blocksize;       // 块大小
    int (*d_open)(struct device *dev, uint32_t open_flags);
    int (*d_close)(struct device *dev);
    int (*d_io)(struct device *dev, struct iobuf *iob, bool write);
    int (*d_ioctl)(struct device *dev, int op, void *data);
};
```
所有具体的设备（它、disk0、stdin等）都实现并填充这个结构体，VFS 层通过这个统一接口与设备交互。

### 5. open 流程
1.  **用户态**：调用 `open(path, flags)`。
2.  **系统调用**：进入内核 `sys_open` -> `sysfile_open`。
3.  **VFS 层**：
    *   `sysfile_open` 拷贝路径字符串。
    *   调用 `file_open`：分配一个空的 `struct file` 和文件描述符 fd。
    *   调用 `vfs_open(path, flags, &node)`：
        *   `vfs_lookup(path, &node)`：解析路径，由根目录或当前目录逐层查找（调用 `vop_lookup` -> `sfs_lookup`），找到对应的 inode。
        *   如果是创建文件，则调用 `vop_create`。
        *   检查权限。
        *   调用 `vop_open`（对应 `sfs_open` 或设备 open）完成特定打开操作（如截断文件）。
4.  **关联**：`file_open` 将找到的 inode 绑定到 `struct file`，设置 readable/writable 标志。
5.  **返回**：返回 fd 给用户。

### 6. load_icode 和 lab5 中的区别
*   **Lab 5**：用户程序（如 `hello`）是以二进制形式链接在内核镜像中的。`load_icode` 直接通过内存拷贝（`memcpy`）将所谓的“文件”内容从内核内存的一个位置复制到进程的用户空间。
*   **Lab 8**：实现了文件系统。用户程序存储在磁盘（SFS）上。
    *   `load_icode` 接收的是一个文件描述符 fd。
    *   它使用 `load_icode_read` 函数，该函数内部调用 `sysfile_read` -> `file_read` 从磁盘读取 ELF Header 和 Program Header。
    *   在加载段数据时，它是从磁盘 `read` 到分配的页面中，而不是单纯的内存拷贝。
    *   这更接近真实的操作系统加载程序的过程。

### 7. 终端程序怎么实现的
终端由 `stdin` (输入) 和 `stdout` (输出) 两个设备组成（位于 `kern/fs/devs/`）。
*   **stdin** (`dev_stdin.c`)：
    *   维护一个循环缓冲区 `stdin_buffer` 和一个等待队列 `wait_queue`。
    *   中断处理例程（`dev_stdin_write`）在键盘输入时被触发，将字符写入缓冲区并唤醒等待的进程。
    *   读操作（`dev_stdin_read`）从缓冲区读取，如果缓冲区空则睡眠等待（`schedule()`）。
*   **stdout** (`dev_stdout.c`)：
    *   写操作（`stdout_io`）直接调用 `cputchar` 输出到串口/屏幕。
    *   通常不涉及缓冲区或阻塞（在简单的实现中）。

### 8. 读取文件和目录的区别
*   **读取文件** (`read`)：读取的是文件内容的字节流。用户看到的只是数据。
*   **读取目录** (`getdirentry`)：
    *   在底层 SFS 中，目录也是一种文件，其内容是一系列 `sfs_disk_entry` 结构（inode号 + 文件名）。
    *   但在系统调用接口层面，通常不直接用 `read` 读取目录内容（得到乱码般的结构体字节）。
    *   使用 `sysfile_getdirentry`，它调用 `vop_getdirentry`。文件系统会解析目录的底层数据结构，将有效的目录项填入标准的 `struct dirent` 返回给用户。
    *   区别在于：目录读操作需要解析特定的文件系统目录格式。

### 9. 管道，进程通信，进程读写互斥，缓冲区
*   **管道 (Pipe)**：一种最基本的 IPC 机制。本质上是内核中的一个缓冲区。
*   **进程通信**：Pipe 允许一个进程写，另一个进程读，实现数据单向流动。
*   **缓冲区**：数据存储在内存页或循环队列中。写者写满时阻塞，读者读空时阻塞。
*   **互斥**：需要自旋锁或信号量保护缓冲区，防止读写指针（read_pos, write_pos）并发竞争导致数据错乱。
    *   ucore 中需要实现 `pipe_inode`，其 read/write 操作对应对缓冲区的存取，并结合 `wait_queue` 实现同步。

### 10. 文件系统 read 系统执行流程（从用户接口层往下）
1.  **用户调用**：`read(fd, buffer, length)`
2.  **Syscall**：触发 `SYS_read`，进入内核 `sys_read`。
3.  **Sysfile**：调用 `sysfile_read`。
    *   检查 fd 合法性。
    *   分配临时内核缓冲区（防止用户非法内存访问）。
    *   调用 `file_read`。
4.  **File 层**：`file_read`
    *   根据 fd 找到 `struct file`。
    *   获取当前文件偏移量 `pos`。
    *   构造 `iobuf`。
    *   调用 `vop_read(file->node, iob)`。
5.  **VFS/Inode 层**：`vop_read` 宏展开为 `inode->in_ops->vop_read`，即 `sfs_read`。
6.  **SFS 层**：`sfs_read` -> `sfs_io` (加锁) -> `sfs_io_nolock`。
    *   `sfs_io_nolock` 将读取请求分解为若干块（Block）。
    *   对于每一块，调用 `sfs_bmap_load_nolock` 将文件逻辑块号映射为磁盘物理块号。
    *   调用 `sfs_rbuf`（读入 buffer）或 `sfs_rblock`。
7.  **Device 层**：`sfs_rbuf` 调用 `dop_io` -> `disk0_io` (或 `ide_io`)。
    *   驱动程序向磁盘控制器发送命令读取扇区。
    *   数据 DMA 到内存。
8.  **返回**：调用链逐层返回，`sysfile_read` 将内核缓冲区数据 copy 到用户 buffer，更新 fd 的 offset。

## 扩展练习 Challenge 1：UNIX PIPE 机制设计方案

### 1. 数据结构设计
需要在内核中定义专门用于管道的数据结构，可以复用 inode 机制，创建一个特殊的 inode 类型 `device` 或 `pipe`。

```c
// 管道缓冲区大小
#define PIPE_SIZE 4096

// 管道主要结构
struct pipe_inode_info {
    mutex_t mutex;              // 互斥锁，保护缓冲区
    wait_queue_t wait_reader;   // 读等待队列（缓冲区空时等待）
    wait_queue_t wait_writer;   // 写等待队列（缓冲区满时等待）
    unsigned int nreaders;      // 读者计数
    unsigned int nwriters;      // 写者计数
    unsigned int head;          // 缓冲区头指针（写位置）
    unsigned int tail;          // 缓冲区尾指针（读位置）
    char buffer[PIPE_SIZE];     // 环形缓冲区
};

// 也是一种特殊的 inode
struct pipe_inode {
    struct inode inode;         // 继承 inode
    struct pipe_inode_info *pipe_info;
};
```

### 2. 接口设计
*   **系统调用**：`int pipe(int fd[2])`
    *   创建两个 `file` 结构和两个 fd。
    *   `fd[0]` 用于读，`fd[1]` 用于写。
    *   创建一个 `pipe_inode_info` 实例。
    *   将两个 `file` 都指向同一个 `pipe_inode`，但 open_flags 不同（一个是 O_RDONLY，一个是 O_WRONLY）。

### 3. 同步互斥处理
*   **读操作 (`pipe_read`)**:
    1.  获取 `mutex`。
    2.  循环检查：如果缓冲区为空 (`head == tail`)：
        *   如果 `nwriters == 0` (没有写者了)，返回 0 (EOF)。
        *   否则，释放 `mutex`，在 `wait_reader` 上睡眠。
        *   被唤醒后重新获取 `mutex` 并检查。
    3.  读取数据，更新 `tail`。
    4.  如果读取前缓冲区是满的，现在有空间了，唤醒 `wait_writer`。
    5.  释放 `mutex`。
*   **写操作 (`pipe_write`)**:
    1.  获取 `mutex`。
    2.  循环检查：如果缓冲区满 (`(head + 1) % SIZE == tail`)：
        *   如果 `nreaders == 0` (没有读者了)，发送 SIGPIPE 信号或返回错误（Broken Pipe）。
        *   否则，释放 `mutex`，在 `wait_writer` 上睡眠。
    3.  写入数据，更新 `head`。
    4.  如果写入前缓冲区是空的，现在有数据了，唤醒 `wait_reader`。
    5.  释放 `mutex`。

## 扩展练习 Challenge 2：UNIX 软连接和硬连接机制设计方案

### 1. 硬连接 (Hard Link)
硬链接意味着多个文件名（目录项）指向同一个 inode。

*   **数据结构**：现有的 `sfs_disk_inode` 已经包含 `nlinks` 字段。
*   **接口**：`int link(const char *oldpath, const char *newpath)`
*   **实现概要**：
    1.  解析 `oldpath` 找到对应的 inode (`target_node`)。
    2.  检查 `target_node` 是否为目录（通常不允许对目录做硬链接以防环路）。
    3.  解析 `newpath` 的父目录。
    4.  在 `newpath` 父目录中创建一个新的 dirent（目录项）。
    5.  设置新 dirent 的 `ino` 为 `target_node->ino`。
    6.  **关键**：`target_node` 的磁盘 inode 中的 `nlinks` 加 1。
    7.  将 inode 更改刷回磁盘（`sfs_wbuf`）。
*   **删除 (Unlink)**：
    1.  当 `unlink` 一个文件时，删除目录项。
    2.  对应 inode 的 `nlinks` 减 1。
    3.  只有当 `nlinks == 0` 且打开计数为 0 时，才真正释放 inode 的数据块（reclaim）。

### 2. 软连接 (Soft Link / Symlink)
软链接是一个特殊的文件，其内容是指向另一个文件的路径字符串。

*   **数据结构**：
    *   `sfs_disk_inode` 的 `type` 字段增加一种类型 `SFS_TYPE_LINK`。
*   **接口**：`int symlink(const char *target, const char *linkpath)`
*   **实现概要**：
    1.  创建一个新文件，类型为 `SFS_TYPE_LINK`。
    2.  将 `target` 路径字符串作为文件内容写入该文件的数据块。
    3.  `readlink` 系统调用：读取该文件的内容返回给用户。
*   **Open 处理**：
    *   修改 `vfs_lookup` 流程。如果在路径解析过程中遇到了 `SFS_TYPE_LINK` 类型的 inode：
        *   读取其内容（目标路径）。
        *   如果引用计数未超过限制（防止死循环），则用目标路径替换当前解析路径，继续查找。
